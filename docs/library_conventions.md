# ライブラリ実装規約

この文書は、個別のアルゴリズムに依存しない、提出用ライブラリ共通の実装方針を定める。
新しい部品を追加するときは、機械的な統一よりも「コンテスト中に短く、迷わず使えること」を優先する。

## 名前と配置

- 物理ファイルは`cp/`以下に置くが、`cp`ディレクトリ自体をinclude rootとする。
  利用側とヘッダ内部では`#include "graph/graph.hpp"`のように先頭の`cp/`を書かない。
- フォルダ名には、カテゴリを表す英単語を一語だけ使う。
- 頭文字だけの略語は、初見で意味を復元しにくいためフォルダ名に使わない。
- 複数語をすべて連結して長くするのではなく、意味を代表する一語を抜き出す。
  例としてdata structureは`structure`へ置く。
- `utility`は、既存の汎用構造を問題文の操作名で直ちに使える完成品に限って置く。
  無関係な小物を集める`misc`や`other`の代用にはしない。
- `recipe`は、複数の汎用部品へ狭い問題族の定型遷移まで加えた高水準shortcutに使う。
  recipeは明示的にincludeし、カテゴリ別・全体の`all.hpp`へは入れない。
- 特定の一問にしか説明できない完成解法はrecipeにも置かない。下位の再利用部品だけを残し、
  問題固有部分は`verify`側へ書く。
- 公開する型名は、`fenwicktree`のように原則として小文字にする。
  大文字入力のためのShift操作を減らし、タイプミスを避けるためである。
- `Mo`のように、競プロで定着した非常に短い名前は慣用表記を使ってよい。
- 関数とメソッドは小文字を基本とし、複数語では必要に応じてsnake_caseを使う。

## 公開API

- データ構造の公開型は、原則として`struct`で定義する。
- 同じ概念の操作は、引数の個数によるオーバーロードで一つの名前へまとめる。
  例えば累積和と区間和は`sum(right)`と`sum(left, right)`にする。
- 覚える名前と入力文字数を減らせるなら、説明的な長い名前を機械的に増やさない。
- 配列として扱うデータ構造には、意味のある要素取得を`operator[]`などで用意する。
  内部表現上、参照を返せない場合は値を返し、そのことを文書に記載する。
- 更新可能な配列型には、構造上実現できる限り`add`だけでなく`set`も用意する。
- 公開型と公開関数・メソッドの直前には、補完候補で読める短い使用例コメントを書く。
- 提出ヘッダ内の説明コメントは日本語で書く。識別子と呼び出し例はコードどおりに記載する。
- 計算量を定義できる公開操作は、補完用コメントへ時間計算量をできるだけ記載する。
  追加領域が重要な操作では空間計算量も記載する。
- `[[nodiscard]]`のような補助属性は必須にしない。実戦コードの見通しと効果を比較して採用する。

## 実戦機能と前提

- あらゆる入力で使える機能だけに限定しない。よく使う制約付き機能も積極的に収録する。
- 制約付き機能を省く代わりに、その前提を宣言直前のコメントとAPI文書へ明記する。
- Fenwick Treeの非負列に対する二分探索のように、構造を利用して高速化できる定番操作は提供する。
- 汎用性だけを理由に実戦的なshortcutを削らない。一方、誤用条件、返り値、見つからない場合の
  挙動は具体的に定義する。
- ライブラリ内部の美しさだけでなく、利用側の提出コードを最小にすることを優先する。
  定型的な列挙、状態集約、不変条件の維持はライブラリへ入れ、汎用部品の上に実戦向けの
  高水準shortcutを用意する。
- 短く書けることだけを目的に、特定問題の解答を名前だけ変えて関数化しない。同じ制約構造を持つ
  他の問題へ適用できる抽象化であり、そのうえで利用側の実装量を減らせることを収録条件とする。
- 主要な抽象部品が揃ったカテゴリでは、類似関数を増やすより先に、選択基準、適用条件、典型的な誤用、
  検証状況を文書へ追加する。

## 型による設計

- オブジェクトの生成後に変わらない重要な性質は、実行時のboolではなく型で表現する。
  例えばグラフの有向・無向、重みあり・なしは公開型から判別できるようにする。
- 利用者には`directed_graph`と`undirected_graph`のような、意味を直接読める公開型を用意する。
  `graph<true, false>`のように意味を思い出す必要がある指定は公開APIに使わない。
- 性質ごとに公開型を分けても、同じ処理を別々に実装しない。共通部分は`detail`内の実装へまとめ、
  公開型はaliasまたは薄いwrapperとして提供する。
- 両方の型で同じアルゴリズムを使える場合は、一つのtemplate関数とconceptで実装する。
  型ごとに同じ関数を複製しない。
- 特定の性質を必要とするアルゴリズムは`requires`で制約し、誤用をコンパイル時に拒否する。
  例えばトポロジカルソートは有向グラフ、最小全域木は無向グラフ、Dijkstraは重み付きグラフに限る。
- 型で表すのは、誤用防止の効果が高く、通常は生成後に変化しない性質に限る。
  問題ごと・操作ごとに変化する細部まで型へ押し込み、呼び出しやエラーメッセージを複雑にしない。

## レビュー時の確認

- 名前を見ただけで用途を思い出せるか。
- Shiftや長い識別子による入力負担を避けているか。
- 同じ概念へ複数のメソッド名を覚える必要がないか。
- 配列らしい取得と、実戦で必要な更新方法が揃っているか。
- 制約付きでも有用な定番操作を、安全性だけを理由に落としていないか。
- 利用側に毎回同じ列挙や状態管理を書かせていないか。
- APIが特定問題の入出力ではなく、他の問題にも現れる制約や操作を表しているか。
- 高水準shortcutなら、coreではなくrecipeへ置くべき狭さではないか。
- エディタの補完表示だけで、基本的な呼び出し方と重要な前提を確認できるか。
- 説明コメントが日本語で簡潔に理解できるか。
- 補完用コメントから主要操作の計算量を確認できるか。

## API文書

- `all.hpp`を除く各`cp/<category>/<name>.hpp`に、同名の`docs/<category>/<name>.md`を一つ対応させる。
- 文書は「そのページだけでheaderを正しく使えること」を完成条件とする。実装方針の説明だけで終わらせず、
  publicな型・関数・コンストラクタ・メソッドをすべて列挙する。
- ACLの文書構成を基準に、各public APIについて次を具体的に書く。
  1. 実際のC++宣言
  2. 何を行うか
  3. 各引数の意味、添字・区間、単位
  4. 返り値と「見つからない場合」などの境界時の値
  5. template引数・値型に必要な演算
  6. 入力制約・使用前提
  7. 時間計算量と、重要なら追加領域
- ページ冒頭には、用途、include、最小の呼び出し例を書く。末尾には頻出事故、実在問題の使用例、
  judge ACまたは未確認の検証状態を書く。
- 同じ名前のoverloadは引数リストごとに分けて説明する。`sum(right)`と`sum(left,right)`を
  一項目へ曖昧にまとめない。
- 区間の閉開、0-index/1-index、根や始点の既定値、破壊的変更の有無、返り値が参照か値かを省略しない。
- 全ページはoj-verifyのFront Matterとして`documentation_of: //cp/...`を持つ。
  `python3 tools/docs.py sync`でFront Matterとheaderコメント由来の`APIリファレンス`を同期する。
  自動生成範囲は直接編集せず、その前の手書き部分で背景・引数・制約・事故例を補足する。
- 変更後は`python3 tools/docs.py check`を実行する。headerの公開コメントや宣言を変えた場合も、
  docsを変更していないからと省略せず、`sync`して差分を確認する。
- 自明な集約専用ヘッダを除き、実在する公開問題に結び付けた使用例を書く。説明用に架空の問題を作らない。
- judgeでACを完走した例と、問題候補だけを確認した未検証例を明確に区別する。
- 適切な公開問題を見つけられない場合は、無理に例を作らず「問題未選定」と記録する。

## 検証の進め方

ABCの過去問監査は、各回のG問題以降とその回の最終問題を最優先する。重い状態管理や高度な典型が
現れやすく、再利用可能なcore・recipeの不足を先に発見できるためである。G以降を分類・実装してから、
C〜Fを穴埋めする。同じ回にGより後の問題がある場合は、その最終問題を先に確認する。

- `verify/*.cpp`は提出コードとしての読みやすさと転用しやすさを優先し、原則として
  `using namespace std;`と`using namespace poe;`を使う。ライブラリヘッダ内では使わない。
- `verify/*.cpp`は、このリポジトリのライブラリを実際に使う問題にだけ作る。問題固有解だけで
  完結し、ライブラリを一つも使わない問題はcoverage表へ方針を記録するだけに留める。
- verifyコードが標準ライブラリだけを使う場合は`using namespace poe;`も書かない。

検証は、変更箇所へ近いものから順に段階的に行う。同じ内容を複数のコマンドで重ねて
確認せず、失敗した段階だけを詳しく調べる。

1. 変更したヘッダごとに`python3 tools/dev.py check <header> --profile quick`を一度実行する。
   この中にlint、依存関係、単体include、二重include、ODR、usage test、bundle検査が含まれるため、
   理由がなければ同じcompileを個別に繰り返さない。
2. アルゴリズムの意味を変更した場合だけ、対応するunit testまたは愚直解とのproperty testを実行する。
   無関係なカテゴリや、変更していないflowなどのtestは実行しない。
3. 公開問題に直接対応する実装だけ、登録済みの`verify/*.test.cpp`を`oj-verify`で検査する。
   代表問題が同じ性質を十分に覆うなら、似た問題を機械的に増やさない。
4. `all.hpp`、metadata、依存関係を変更したときは、最後に全体lintと依存関係検査を一度だけ行う。
5. sanitizer、最大ケース、長時間stressは、内部表現や境界条件を変更した場合、またはstable昇格時に行う。

LLMを介した作業では、成功時の長い出力を会話へ流さない。完全なログは`/tmp`へ保存し、
終了コードと短い要約だけを確認する。失敗した場合に限り、そのログの末尾や該当箇所を読む。
JSONの構文確認には内容を全表示する`python3 -m json.tool`ではなく、次のような無出力の検査を使う。

    python3 -c 'import json; json.load(open("library.json"))'

`oj-verify`などの詳細ログも同様に一時ファイルへ保存する。最終報告には、実行した検査名、
対象、成否、公式ケース数だけを残し、ケースごとの成功ログは転載しない。
