# DPライブラリ選択ガイド

DPカテゴリの主要な抽象部品は一通り揃ったため、今後は問題数に合わせて専用関数を増やすのではなく、
既存部品の適用条件、組み合わせ方、検証例を改善する。この文書は問題名ではなく、状態と遷移の形から
使うヘッダを選ぶための入口である。

## 抽象部品・標準アルゴリズム・recipe

- **抽象部品**は、利用側が状態や遷移を定義する。`digitdp`、`frontierdp`、`aliens_min`など。
- **標準アルゴリズム**は、ナップサック、LCS、TSPのように入力模型と答えが広く共有される完成品。
- **recipe**は、複数の抽象部品に狭い問題族の定型遷移まで足したもの。便利だが標準APIではなく、
  `all.hpp`からincludeしない。

「問題文の入力をほぼそのまま受け取り、その一問の答えを返す」だけで、別問題に共通する状態・制約を
説明できないものは追加しない。短さだけでなく、別問題へ持ち運べることを収録条件にする。

## 最適化を使う前の確認

1. 愚直DPの状態と遷移を書き、答えの意味を固定する。
2. 計算量を支配する軸を確認する。`n`だけでなく容量、価値総和、幅、状態数を見る。
3. 単調性・凸性・Monge性など、最適化が要求する条件を証明する。
4. 同値な最適解のtie breakを確認する。Aliens trickでは個数の選び方が結果に影響する。
5. `inf + cost`、総和、罰金積のoverflowを見積もる。

## よくある選択ミス

- 0/1、個数無制限、個数上限付きナップサックのループ方向を混同する。
- `leadingzero::ignore`で数0が`started=false`になる点を忘れる。
- 最適遷移位置の単調性をサンプルだけで仮定してdivide-and-conquer最適化を使う。
- frontier DPで状態を正規化せず、同型状態を別々に保持する。
- recipeの固定遷移を、禁止マスや辺選択のある問題へ流用する。

## 検証状況の読み方

各ヘッダ文書には確認済みの事実だけを書く。未提出・未確認という作業状況は利用者向けdocsへ書かない。

全問題の分類と「ライブラリにしない」判断は[EDPC・TDPC・NDPC分類](catalog.md)を参照する。
